AGENT LEARNING · ARCHITECTURE

法律咨询 Agent · 项目架构图

学习型 Python Agent 项目：本地 Web UI 咨询入口，一轮咨询由 4 次 LLM 调用编排， 本地法条 RAG（BGE-M3 + Chroma）与公网案例检索并行取证，带会话持久化、跨会话案件记忆与链路自修复。

Python · FastAPI React 19 + Material UI（Vite 构建） OpenAI 兼容 Responses API BGE-M3 + Chroma 博查 Web Search unittest + Vitest

⇅ NDJSON 事件流 / REST ⇅

后端 web_app/server.py（FastAPI）

  • GET·DELETE /api/sessions 历史会话；GET /api/metrics 轮级指标聚合

  • 轮首按关键词召回跨会话记忆，轮末写会话快照与案件记忆
法律咨询业务编排层 agent_system/legal_consultation/

session.py · LegalConsultationSession

串联一轮咨询 workflow；StreamingAnswerSanitizer 流式答案行级清洗； export / restore_snapshot 会话快照导出与恢复。

subtasks.py · 四个子任务

  • LegalCaseStateUpdater 案件状态更新
  • LegalCaseRagSubtask 案情拆解 + 多 query 法条检索
  • LegalDeterministicWebSearchSubtask 确定性公网检索
  • LegalCaseAnalyzer 综合分析（风险 + 目录 + 下一步）

models.py / factory.py

LegalCaseState 等结构化模型；会话工厂让 LLM client 与 BGE-M3 检索器跨会话共享、惰性构建。

链路自修复：状态更新 / RAG / 综合分析失败时发 legal_selfheal 并降级继续；最终回答失败自动非流式重试一次。 每轮结束发 legal_turn_metrics（阶段耗时 · LLM usage · 自修复次数）。
通用能力层 agent_system/

agent/ 工具调用闭环

  • AgentRunner 通用 function-call 循环（不含业务逻辑）
  • ToolRegistry / LocalTool / AgentEvent
  • 工具：search_legal_articles、search_legal_articles_by_keyword、web_search

llm/ 模型访问

  • OpenAIChatClient 封装 Responses API（流式 + usage 累计）
  • ChatSession 普通聊天历史 · factory 构建入口

planning/ 检索规划

LegalQueryPlanner 把复杂案情拆成多组检索 query 与关键词；只做拆解，不产出法律结论。

retrieval/ 法条 RAG

  • LocalBGEEmbeddingModel（BGE-M3 本地推理）
  • LegalChromaStore 向量库管理
  • LegalArticleRetriever：语义检索 + 关键词精确匹配兜底

storage/ · memory/ 落盘

  • SessionStore 会话存档（meta / snapshot / events）
  • MemoryStore 跨会话案件记忆（关键词召回）
数据与外部服务 data/ · 第三方 API

本地数据 data/

  • 最核心法条_9k.json 法条原始数据
  • chroma/ 向量库持久化目录
  • sessions/ 会话存档 · memory/ 案件记忆

外部服务

  • OpenAI 兼容 LLM API（Responses 协议）
  • 博查 Web Search API（公网案例 / 司法实践）
  • BGE-M3 模型（本地推理，首次运行自动下载）

配置 agent_system/config.py

读取优先级：环境变量 > config_local.py（不入 git）> 内置默认值。 涵盖 LLM / Embedding / Chroma / 检索 / Web Search 配置。

一轮咨询执行流程

成功链路共 4 次 LLM 调用；右侧为并行泳道与暂停旁路
▶ 用户发送案情 / 补充信息 浏览器

POST /api/chat（携带 session_id，为空则新建会话并回传 session 事件）。 轮首后端按关键词召回其他会话的案件记忆，注入本轮 prompt。

legal_memory_recalled
1 案件状态更新 LLM 调用 ①

LegalCaseStateUpdater 把新对话合并进结构化案件状态，并给出可补充的关键信息建议。

case_state_updated legal_missing_details_suggested
状态更新完成后，立即后台启动公网检索 →
2 检索 query 规划 LLM 调用 ②

LegalQueryPlanner 把案情拆成多组语义 query 和精确关键词，并推荐目标法律名。

∥ 并发本地法条检索（RAG） 本地检索

语义检索（BGE-M3 + Chroma）∥ 关键词精确匹配兜底，两路 fan-out 并发执行； 证据按 evidence_rank_score 融合重排（多 query 重复命中与规划词微调分数）。

legal_rag_query_started legal_case_rag_done
3 案情综合分析 LLM 调用 ③

LegalCaseAnalyzer 一次结构化调用同时产出：风险识别 + 案情目录 + 下一步动作（对外仍按三个事件发出）。

legal_risk_analyzed legal_analysis_catalog_built legal_next_action_decided
等待公网检索 future 汇合，整理右侧参考资料栏数据
4 最终回答（流式） LLM 调用 ④ · 流式

AgentRunner 以空工具注册表流式生成（最后阶段不再开放检索工具）；增量先过 StreamingAnswerSanitizer 行级清洗再推送。回答固定律师口吻章节：结论 / 当前关键点 / 法律风险 / 现在该做什么 / 还缺哪些关键信息。

legal_reference_materials answer_delta message_done
✓ 轮末提交与落盘 持久化

SessionStore 原子写快照并追加 turn_committed（含白名单轮级指标）； MemoryStore 沉淀案件记忆供后续会话召回。失败轮只记 turn_failed 并回滚会话内存，不写快照。

legal_turn_metricsdone
旁路 暂停等待补充

← 由步骤 ① 判定 should_pause_for_supplement

发 legal_supplement_required，后端输出顶层 pause 事件（无 final）； 前端弹出补充信息弹窗。本轮作为成功轮提交案件状态与追问历史，用户补充后下一轮继续完整链路。

并行 确定性公网检索（后台 future）

← 步骤 ① 完成后立即启动，与 RAG、综合分析全程并行

  • 复用 web_search 工具，3 条固定 query 并发：相似案例 / 司法解释与权威规定 / 裁判规则与司法实践
  • 司法解释 query 用 include 限定官方与专业法律站点，其余用 exclude 排除低质站点
  • 结果按域名权威度分级重排（权威 / 专业 / 普通 / 低置信），每条 query 取 10 选 5
  • 失败只转 warning，不中断本地法条分析
  • 结果在步骤 ④ 前汇合，经白名单进入右侧参考资料栏 ↘
legal_web_search_started legal_web_search_done
自修复 失败降级与重试
  • ①②③ 任一失败：发 legal_selfheal（degraded），以旧状态 / 空证据 / 默认动作降级继续
  • ④ 失败：自动非流式重试一次（retried），再失败才整轮回滚
  • 每轮发 legal_turn_metrics：阶段耗时、LLM usage 增量、自修复次数

图 3

持久化与观测

工具调用过程不进公开消息历史，观测数据全部走事件通道

会话存档 data/sessions/<id>/

  • meta.json 标题 / 创建·更新时间 / 轮次
  • snapshot.json 公开 messages + case_state + 资料 + 待补充请求（原子写）
  • events.jsonl append-only：session_created / turn_committed / turn_failed（单调 seq）
  • session_id 白名单正则校验，防路径穿越

跨会话记忆 data/memory/

  • 轮末把 case_state 知识沉淀为案件记忆
  • 新会话轮首按关键词召回（排除本会话自身），注入状态更新与最终回答 prompt
  • 命中时发 legal_memory_recalled，写盘失败降级为非致命提示

运行指标 GET /api/metrics

  • 每轮 legal_turn_metrics 过白名单后随 turn_committed 落盘
  • 接口扫描各会话 events.jsonl 聚合：轮数 / 失败轮 / 成功率 / 总耗时 / LLM usage
  • 不维护内存计数器，重启不丢数据
