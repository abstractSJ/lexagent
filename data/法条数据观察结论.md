# 法条数据观察结论

日期：2026-06-30

本文档记录 `最核心法条_9k.json` 的数据结构、内容分布、清洗建议，以及后续使用 Chroma 构建法条向量数据库时的推荐 schema。该文档可作为后续 RAG / Agent 工具接入的设计依据。

---

## 1. 数据文件概览

数据文件：`data/最核心法条_9k.json`

整体结构是一个 JSON 数组，每条记录也是一个长度为 3 的数组：

```json
[
  ["分类", "标题或法律名称", "正文片段"],
  ["分类", "标题或法律名称", "正文片段"]
]
```

统计结果：

| 项目 | 结果 |
|---|---:|
| 顶层类型 | list |
| 总记录数 | 9,671 |
| 每条记录字段数 | 固定 3 个 |
| 三个字段类型 | 全部为字符串 |
| 唯一分类数 | 10 |
| 唯一标题数 | 103 |
| 文件大小 | 约 4.19 MB |

字段含义：

| 下标 | 推荐字段名 | 含义 |
|---:|---|---|
| 0 | `category` | 分类、法律部门或来源，例如“民法商法”“社会法”“公安部”等 |
| 1 | `title` | 法律、法规、规章、分编或资料标题，部分标题末尾带日期 |
| 2 | `text` | 正文片段，大多数是单条法条，也混有攻略、标准、前言说明和格式残留 |

## 3. 标题和法律名称分布

唯一标题数：103。

高频标题示例：

| 标题 | 数量 |
|---|---:|
| 合同编 | 531 |
| 刑法 | 509 |
| 公安机关办理刑事案件程序规定2020-07-20 | 389 |
| 刑事诉讼法2018-10-26 | 311 |
| 民事诉讼法2021-12-24 | 292 |
| 海商法1992-11-07 | 278 |
| 公安机关办理行政案件程序规定2019-04-27 | 267 |
| 物权编 | 259 |
| 证券法2019-12-28 | 226 |
| 公司法2018-10-26 | 218 |

标题特点：

1. 很多标题末尾带 `YYYY-MM-DD` 日期，例如 `公司法2018-10-26`。
2. 入库时应拆分为：
   - `legal_name`: 去掉日期后的法律名称，例如 `公司法`
   - `version_date`: 标题末尾日期，例如 `2018-10-26`
3. “民法典”分类下存在 `合同编`、`物权编`、`总则` 等分编标题。单独使用这些标题语义不完整，建议在构造检索文本时补充为：
   - `民法典-合同编`
   - `民法典-物权编`
   - `民法典-总则`

结论：

- 大多数正文很短，符合单条法条或短条款特征。
- 中位数约 92 字，95% 正文不超过 275 字。
- 不适合再粗暴切成 500/1000 字大块；第一版应以“单条原始记录 = 一个向量文档”为主。

## 5. 是否已经按法条切分

总体判断：大部分已经按法条粒度切分，但不是完全干净的法条库。

统计观察：

| 指标 | 数量 |
|---|---:|
| 正文以 `第...条` 开头 | 约 9,419 / 9,671 |
| 正文包含 `第...条` | 约 9,420 / 9,671 |
| 正文中出现 2 个及以上 `第...条` | 约 659 |

说明：

1. 正式法律、法规、规章部分基本是一条记录对应一个法条。
2. 正文中出现多个 `第...条` 不一定是错误，可能是法条内部引用其他条款。
3. 少量记录是前言、公布说明、修订说明、标准条目或格式残留。
4. 有少量正文长度极短或内容异常，需要建库前过滤或标注。

## 6. 连续片段和上下文关系

同一标题下的记录基本连续排列。

这意味着可以保留以下字段，方便命中某条法条后扩展上下文：

- `record_index`: 原始 JSON 顶层数组下标
- `chunk_index_in_title`: 当前标题内部的顺序编号
- `prev_record_index`: 同一标题下的上一条记录下标，可选
- `next_record_index`: 同一标题下的下一条记录下标，可选

适用场景：

- 用户问题需要解释一组连续条文。
- 命中的法条前后条文有定义、例外或程序衔接关系。
- Agent 需要在回答里补充“相关相邻条文”。

第一版检索工具可以提供 `include_neighbors` 参数，默认关闭，需要时返回命中条文前后各 1 条同标题记录。

## 7. 非正式内容和异常内容

数据中存在非正式内容和格式异常，不建议全部混入主法条 collection。

明确存在的非正式或特殊内容：

| 标题 | 数量 | 判断 |
|---|---:|---|
| 劳动仲裁和劳动诉讼的攻略 | 27 | 攻略类、实务类，不是正式法条 |
| 人体损伤程度鉴定标准2014-01-01 | 106 | 标准类条目，不是传统法律条文 |

格式异常示例：

| 异常类型 | 说明 |
|---|---|
| 正文等于 `[` | 明显格式残留，应过滤 |
| 正文以 `[` 开头 | 可能是前言、目录或转换残留 |
| 正文以 `"` 开头或结尾 | 可能是 JSON 字符串切分残留 |
| 正文包含 URL | 多见于攻略类内容 |

建议建库前进行轻量清洗和类型标注，而不是直接全部入库。

## 8. 推荐的 `source_type` 标注规则

建议为每条记录增加 `source_type` 字段：

| source_type | 判断规则 | 建议处理 |
|---|---|---|
| `law_article` | 正文以 `第...条` 开头 | 放入主法条 collection |
| `guide` | 标题为“劳动仲裁和劳动诉讼的攻略”等攻略内容 | 单独 collection 或默认过滤 |
| `standard` | 标题包含“鉴定标准”等标准类内容 | 可单独 collection |
| `preface_or_misc` | 前言、修订说明、公布说明等 | 默认不进入主法条库 |
| `artifact` | 明显格式残留，例如正文只有 `[` | 过滤 |

主法条向量库第一版建议只放：

```text
source_type == "law_article"
```

## 9. 推荐的 Chroma collection 设计

主 collection 名称建议：

```text
legal_articles_v1
```

用途：存放正式法条。

如果后续需要，也可以扩展：

```text
legal_guides_v1
legal_standards_v1
```

分别存放实务攻略和标准类材料，避免与正式法条混淆。

## 10. 推荐的 document text 构造方式

不建议只使用原始正文 `text` 入向量库，因为单条法条往往缺少法律名称上下文。

推荐格式：

```text
分类：{category}
法律名称：{legal_name}
标题：{title}
版本日期：{version_date}
条文：{article_no}

{text_clean}
```

更短格式：

```text
{legal_name}（{version_date}）{article_no}
{text_clean}
```

民法典分编建议特殊处理：

```text
民法典-{title} {article_no}
{text_clean}
```

原因：`合同编`、`物权编`、`总则` 这类标题单独看语义不完整，补充 `民法典` 可以提升中文语义检索质量。

## 11. 推荐的 Chroma metadata schema

每条向量文档建议 metadata 使用标量字段：

```python
{
    "source_path": "data/最核心法条_9k.json",
    "record_index": 1234,
    "category": "社会法",
    "title": "劳动合同法2012-12-28",
    "legal_name": "劳动合同法",
    "version_date": "2012-12-28",
    "article_no": "第三十八条",
    "chunk_index_in_title": 37,
    "source_type": "law_article",
    "char_len": 156,
    "is_article": True,
    "has_format_artifact": False,
    "text_hash": "..."
}
```

字段说明：

| 字段 | 用途 |
|---|---|
| `source_path` | 原始数据文件路径 |
| `record_index` | 原始 JSON 顶层数组下标，方便回溯 |
| `category` | 分类或法律部门 |
| `title` | 原始标题，保留日期 |
| `legal_name` | 去掉日期后的法律、法规或规章名称 |
| `version_date` | 标题末尾日期，没有则为空字符串 |
| `article_no` | 提取出的条号，例如 `第三十八条` |
| `chunk_index_in_title` | 同一标题下的顺序编号 |
| `source_type` | 内容类型，例如 `law_article`、`guide`、`standard` |
| `char_len` | 清洗后正文长度 |
| `is_article` | 是否以 `第...条` 开头 |
| `has_format_artifact` | 是否存在明显格式残留 |
| `text_hash` | 清洗后文本 hash，用于稳定去重和增量更新 |

Chroma `id` 建议稳定生成：

```text
law:{record_index}:{text_hash前8位}
```

这样可以重复构建、增量 upsert，并避免随机 ID 导致重复入库。

## 12. Chunking 策略

### 第一版原则

```text
一条原始记录 = 一个 Chroma document
```

原因：

1. 当前数据大多数已经是单条法条粒度。
2. 法律回答需要引用具体法条。
3. 过大的 chunk 会混淆多个法条，降低可引用性。
4. 原始正文中位数很短，不需要再切小。

### 长文本处理

对于超过 800～1000 字的极长记录，可以做特殊处理：

1. 按自然段或句号切分为多个子块。
2. metadata 保留：
   - `parent_record_index`
   - `chunk_part`
   - `parent_article_no`
3. 检索返回时优先补全父条文，避免只返回半句。

第一版也可以暂不拆长文本，只保留原文，后续根据检索效果再优化。

## 13. Embedding 模型建议

当前项目的 OpenAI-compatible LLM 接口可以调用模型列表，但实测没有可用的 `/v1/embeddings` endpoint 或 embedding 模型。

因此有两个可行路线：

### 方案 A：官方 OpenAI embedding

可选模型：

```text
text-embedding-3-small
text-embedding-3-large
```

优点：

- 接入简单
- 稳定性较高
- 与 OpenAI SDK 兼容

缺点：

- 法条内容会发送到外部 API
- 需要额外 API key 或官方 OpenAI 可用环境
- 有调用成本

### 方案 B：本地中文 embedding

推荐模型：

```text
BAAI/bge-m3
```

优点：

- 中文语义检索效果较好
- 数据不出本地
- 9,671 条法条规模不大，本地建库完全可行
- 查询阶段没有额外 API 成本

缺点：

- 需要安装本地模型依赖
- 首次下载模型和建库会较慢
- 需要管理本地运行环境

当前更推荐第一版使用本地 BGE-M3 构建 Chroma，然后继续用现有 LLM 接口生成最终回答。

## 14. 检索工具设计建议

给 Agent 暴露一个工具：

```text
search_legal_articles
```

参数建议：

```python
{
    "query": "用户法律问题或检索关键词",
    "top_k": 5,
    "legal_name": "可选，指定法律名称",
    "category": "可选，指定分类",
    "article_no": "可选，指定条号",
    "source_type": "law_article",
    "include_neighbors": False
}
```

返回结构建议：

```python
{
    "ok": True,
    "query": "公司不签劳动合同怎么办",
    "results": [
        {
            "rank": 1,
            "id": "law:1234:abcd1234",
            "score": 0.82,
            "category": "社会法",
            "legal_name": "劳动合同法",
            "title": "劳动合同法2012-12-28",
            "version_date": "2012-12-28",
            "article_no": "第八十二条",
            "text": "用人单位自用工之日起超过一个月不满一年未与劳动者订立书面劳动合同的……",
            "citation": "《劳动合同法》第八十二条"
        }
    ]
}
```

Agent 系统提示词应要求：

1. 法律问题、赔偿问题、程序问题、责任问题、条文依据问题，优先调用 `search_legal_articles`。
2. 不得编造未检索到的法条。
3. 回答中应引用工具返回的 `citation`。
4. 检索不到直接依据时，应明确说明“当前法条库未检索到直接依据”。
5. 输出法律咨询时应提示：内容仅作信息参考，不构成正式法律意见。

## 15. 检索策略建议

第一版建议采用简单稳定的检索流程：

1. 用户问题转 query embedding。
2. 在 `legal_articles_v1` 中向量召回 top 10～20。
3. 默认过滤：

```python
where={"source_type": "law_article"}
```

4. 如果用户明确提到法律名称，例如“劳动合同法”“刑法”“公司法”，可增加 metadata 过滤：

```python
where={"legal_name": "劳动合同法"}
```

5. 如果用户明确提到条号，例如“劳动合同法第八十二条”，优先做精确检索：

```python
legal_name == "劳动合同法"
article_no == "第八十二条"
```

6. 返回给 Agent 的结果控制在 top 5 左右，避免上下文过多导致回答混乱。

## 16. 建库流程建议

第一版建库流程：

1. 读取 `data/最核心法条_9k.json`。
2. 对每条记录提取：
   - `category`
   - `title`
   - `legal_name`
   - `version_date`
   - `article_no`
   - `source_type`
   - `text_clean`
3. 过滤明显无效记录：
   - 空文本
   - 只有 `[` 的格式残留
   - 明显非目标内容
4. 主 collection 只入库：

```text
source_type == "law_article"
```

5. 构造 document text。
6. 批量生成 embedding。
7. 写入 Chroma：
   - `ids`
   - `documents`
   - `metadatas`
   - `embeddings`
8. 在 collection metadata 中记录：
   - `embedding_model`
   - `data_source`
   - `schema_version`
   - `build_time`
   - `source_file_hash`

## 17. 关键注意事项

### Embedding 模型必须固定

建库和查询必须使用同一个 embedding 模型。

如果建库时使用 `BAAI/bge-m3`，查询时也必须用 `BAAI/bge-m3`。

如果后续换成 `text-embedding-3-large` 或其他模型，需要重建整个 Chroma collection。

建议 collection metadata 记录：

```python
{
    "embedding_model": "BAAI/bge-m3",
    "schema_version": "legal_articles_v1",
    "data_source": "data/最核心法条_9k.json"
}
```

查询启动时应检查当前配置的 embedding model 是否与 collection metadata 一致。不一致时直接报错，避免向量空间不一致导致检索结果失真。

### 正式法条和攻略不要混用

主法条库应只放正式法条。

攻略类内容可以以后单独做 collection，并在工具层明确区分：

- 查正式法律依据：`search_legal_articles`
- 查实务攻略：`search_legal_guides`

这样可以降低 Agent 把非正式建议误当作法律依据的风险。

### 返回结果要可引用

法律咨询回答必须能回溯到具体条文。

所以检索结果中必须包含：

- `legal_name`
- `article_no`
- `text`
- `citation`
- `record_index`

推荐引用格式：

```text
《劳动合同法》第八十二条
```

如果没有条号，可引用：

```text
《某某规定》相关条款
```

但应尽量避免没有明确出处的回答。

## 18. 推荐第一版落地方案

第一版目标：让法律咨询 Agent 稳定检索和引用正式法条。

建议实现：

1. 使用本地 embedding 模型 `BAAI/bge-m3`。
2. 使用 Chroma 持久化目录：

```text
data/chroma/
```

3. 创建 collection：

```text
legal_articles_v1
```

4. 每条正式法条作为一个 document。
5. metadata 至少包含：

```text
record_index
category
title
legal_name
version_date
article_no
source_type
chunk_index_in_title
text_hash
```

6. 新增 Agent 工具：

```text
search_legal_articles
```

7. Agent 回答法律问题时先查工具，再基于工具结果回答。
8. 非正式攻略和标准类内容先不混入主库，后续按需要单独 collection。
